# SEO Product Description Generator — Design Spec (AI/LLM Engineer)

## Задача

Разработать ядро (AI-пайплайн) для генерации SEO-описаний товаров. Задание сфокусировано на проверке навыков работы с современными AI-инструментами (LangChain, Zod, Streaming), а не на написании CRUD-бойлерплейта.

**Ожидаемое время выполнения:** 3-4 часа.

---

## Архитектура и стек

Данный микросервис не требует базы данных. Вся логика выполняется *stateless*.

**Обязательный стек:**
- **Node.js / NestJS** (или Express/Fastify по выбору, главное качество архитектуры).
- **LangChain.js**: использование LCEL (LangChain Expression Language), построение цепочек генерации.
- **Zod**: строгая типизация ожидаемого ответа LLM (`Structured Output Parser`).
- **OpenRouter API** (или прямой OpenAI/Anthropic API): интеграция и управление провайдерами моделей.

---

## Функциональные требования

### 1. Эндпоинт генерации (LangChain + Zod)
`POST /api/generate-seo`
- Принимает данные товара (название, категория, массив ключевых слов).
- Строит промпт на основе продвинутого шаблона (Prompt Engineering), интегрируя системный промпт (роль SEO-эксперта).
- Использует **LangChain.js** для вызова LLM.
- Обязует модель вернуть ответ строго по заданной JSON-схеме (используя Function Calling/Structured Outputs валидированные через Zod).
- **Стриминг**: Обеспечивает стриминг результатов по мере генерации (Server-Sent Events) до клиента.

### 2. Структура ответа (Strict JSON / Zod Schema)
AI-пайплайн должен гарантировать возврат объекта со следующей структурой:
- `title` (строка, строго до 60 символов) — кликабельный SEO-заголовок.
- `meta_description` (строка, строго до 160 символов) — продающее мета-описание.
- `h1` (строка) — главный заголовок для карточки.
- `description` (строка) — подробное структурированное описание товара (от 300 символов).
- `bullets` (массив строк) — 3-5 ключевых преимуществ товара маркированным списком.

### 3. Интеграция Flowise
Параллельно с кодом на Node.js, кандидату необходимо собрать аналогичную логику агента в визуальном конструкторе **Flowise**:
- Создать Chatflow/LLM Chain: `Prompt Template` -> `Chat Model` -> `Structured Output Parser`.
- Экспортировать готовый flow в виде JSON-файла (`flowise/seo-chatflow.json`) и вложить в репозиторий.

## Форматы данных эндпоинта

### Request `POST /api/generate-seo`

```json
{
  "product_name": "Кроссовки Nike Air Max 90",
  "category": "Обувь / Спортивная обувь",
  "keywords": ["кроссовки", "Nike", "Air Max", "спортивная обувь", "амортизация"]
}
```

### Response (SSE Stream -> Structured JSON)
В идеале стримятся частичные (partial) JSON-объекты или дельты значений, которые на клиенте собираются в единый валидный объект вида:

```json
{
  "title": "Купить кроссовки Nike Air Max 90 — спортивная обувь",
  "meta_description": "Закажите кроссовки Nike Air Max 90 с уникальной системой воздушной амортизации. Отличный выбор для спорта и города. Доставка и примерка!",
  "h1": "Кроссовки Nike Air Max 90",
  "description": "Подробное текстовое описание...",
  "bullets": [
    "Технология Air Max для идеальной амортизации",
    "Дышащий текстильный верх с кожаными вставками",
    "Прочная гибкая подошва"
  ]
}
```

## Требования к оценке (На что смотрим в первую очередь)

1. **Prompt Engineering**: Насколько глубоко и качественно составлен системный промпт (instruction), чтобы модель соблюдала специфичные лимиты символов (<=60 для title, <=160 для meta) и органично вписывала входные `keywords`.
2. **LangChain.js Mastery**: Архитектура цепочек (Chains/LCEL) внутри Node.js, работа с парсерами вывода, управление контекстом. Умение не просто сделать API call, а построить расширяемый пайплайн.
3. **Надежность (Zod Validations)**: Как кандидат обрабатывает случаи, когда модель «галлюцинирует» поля или ломает JSON структуру. Наличие retry-механизмов.
4. **Стриминг JSON**: Как элегантно решена сложная техническая задача стриминга (полу-валидного в момент стрима) JSON чанков через SSE.
5. **Чистота кода**: Разделение ответственности (роутинг отдельно от AI-сервисов), типизация (TS), обработка таймаутов от LLM провайдера.
